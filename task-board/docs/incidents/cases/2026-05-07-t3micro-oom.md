# 2026-05-07 t3.micro でメモリ不足、SSH 応答不能

**Phase:** 3 (アプリ起動)
**ステータス:** Resolved (commit `8e16c49`)

---

## 症状（Symptoms）

`./start.sh` 実行で Spring Boot 起動を試みた直後、EC2 への SSH が応答しなくなる。

```
ssh: connect to host 54.150.151.67 port 22: Connection timed out during banner exchange
```

AWS API では `state: running` のため EC2 自体は生きているが、システムが thrashing 状態に。

## 兆候（Early Signs）— 次回検知用

- インスタンスタイプが **t3.micro（RAM 1GB）**
- 同居サービスが **Java（Gradle daemon + Spring Boot）+ Node.js（Vite）+ PostgreSQL コンテナ**
- swap 領域が **0B**（`free -h` で確認できる）
- `start.sh` 実行直後に `free -h` で残メモリ < 100MB

## 真因（Root Cause）

t3.micro は物理 RAM 1GB しかない。一方で同時稼働するサービスのメモリ要求合計：

- Gradle daemon ＋ Java compile：~600MB
- Spring Boot 実行時：~500MB
- PostgreSQL Docker：~100MB
- Vite + node_modules：~150MB
- OS overhead：~150MB
- **合計 ~1.5GB**（物理 1GB を超過）

swap が無いため OS が逃がす先がなく、kernel が thrashing → SSH の sshd ですら応答不能になった。

**「無料枠で済ませる」ことを優先した結果、リソース計画が抜けた。**

## 関連パターン

- 「リソース不足」系
- 関連：CPU bound / disk full / inode 不足
- (未抽出) → 将来 `patterns/pattern-resource-shortage.md` として抽象化予定

## 是正処置（Remediation）

`infra/user_data.sh` の cloud-init 段階で **4GB swap ファイル作成**：

```bash
fallocate -l 4G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
echo 'vm.swappiness=10' > /etc/sysctl.d/99-swap.conf
sysctl -p /etc/sysctl.d/99-swap.conf
```

`vm.swappiness=10` で物理メモリ優先・swap は緊急時のみ使用。

## 再発防止（Prevention / Jidoka）

- **同居サービス計画 → メモリ要求合計 → インスタンスサイズ選定** の順を遵守
  - 同居が3つ以上 + Java/Node 系 → 最低 t3.small（2GB）or 4GB swap 必須
- `task-board/docs/version-matrix.md` に **「メモリ要求」列** を追加
- 監視：CloudWatch で MemoryUtilization > 90% でアラート（将来）
- EC2 にデプロイ前 → `free -h` でベースライン確認 → 起動後再確認のルーチン化
