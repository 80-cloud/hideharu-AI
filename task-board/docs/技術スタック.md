## 1. フルスタック構成の全体像

> **「フルスタック」とは**  
> 画面（フロントエンド）・サーバー（バックエンド）・データベースの3層すべてを自分で開発するスタイルのことです。  
> v1.0ではブラウザだけで完結していましたが、v2.0では本格的なWebアプリの構成になります。

### アプリ全体の構造

```
┌─────────────────────────────────────┐
│         ブラウザ（ユーザーの画面）         │
│                                     │
│   React + Tailwind CSS（フロントエンド）   │
│   ・タスクの表示・操作画面                 │
└──────────────┬──────────────────────┘
               │  HTTP通信（Axios）
               │  データをやり取りする
               ▼
┌─────────────────────────────────────┐
│     Spring Boot サーバー（バックエンド）    │
│                                     │
│   Java + Maven で動くWebサーバー         │
│   ・ブラウザからのリクエストを受け取る         │
│   ・タスクの追加・更新・削除の処理をする       │
└──────────────┬──────────────────────┘
               │  JPA/Hibernate（SQL自動生成）
               │  データを保存・取得する
               ▼
┌─────────────────────────────────────┐
│       PostgreSQL（データベース）          │
│                                     │
│   タスクのデータを永続的に保存               │
│   Docker コンテナ内で動作                │
└─────────────────────────────────────┘
```

### v1.0 との比較

| 項目 | v1.0（Vanilla JS） | v2.0（フルスタック） |
|---|---|---|
| データの保存先 | ブラウザの localStorage | PostgreSQL データベース（サーバー） |
| データの消え方 | ブラウザのキャッシュ削除で消える | サーバーに残り続ける |
| 複数端末での利用 | 不可（ブラウザごとに別データ） | 可能（将来的に）|
| サーバー | 不要 | 必要（Spring Boot）|

### APIエンドポイント設計

フロントエンドとバックエンドは「API」という取り決めで通信します。

> **「API」とは**  
> 「このURLにこんなデータを送ったら、こういう結果を返す」という約束事のことです。  
> レストランのメニューのようなもので、「ハンバーガーをください（リクエスト）」→「はい、どうぞ（レスポンス）」のやり取りです。

| HTTPメソッド | URL | 操作 | v1.0の対応関数 |
|---|---|---|---|
| GET | `/api/tasks` | タスク一覧を取得 | `loadTasks()` |
| POST | `/api/tasks` | タスクを新規作成 | `addTask()` |
| PUT | `/api/tasks/{id}` | タスクを更新（編集・カラム移動） | `updateTask()` |
| DELETE | `/api/tasks/{id}` | タスクを削除 | `deleteTask()` |

## 2. バックエンドの採用技術

### 2-1. Java 21（プログラミング言語）

**これは何か：**  
サーバー側（バックエンド）を書くプログラミング言語です。  
「21」はバージョン番号で、**LTS（Long-Term Support＝長期サポート版）** のため、企業でも安心して使えます。

**なぜJavaか：**  
- 日本の企業（特に銀行・保険・SIerなど）での採用率No.1
- 「型」が厳格なので、バグが起きにくく、大規模開発向き
- Spring Boot との組み合わせが業界標準

---

### 2-2. Spring Boot 3.x（Webフレームワーク）

**これは何か：**  
Javaで「Webサーバー」を作るためのフレームワーク（枠組み・土台）です。  
「ゼロからWebサーバーを作る」のは非常に大変ですが、Spring Bootを使うと設定を自動でやってくれるため、開発者はビジネスロジック（タスクの追加・削除などの処理）に集中できます。

> **「フレームワーク」とは**  
> 家を建てるときの「プレハブキット」のようなものです。  
> 骨組みや設備がすでに用意されており、自分は「壁の色」や「部屋の配置」だけを決めれば家が完成します。

**なぜSpring Bootか：**  
- 「Spring Initializr（start.spring.io）」というWebサービスから数クリックでプロジェクトのひな型を生成できる
- 必要なライブラリが「セット」で提供されており、組み合わせを考える手間がない
- 日本でのJava開発の事実上の標準

---

### 2-3. Maven（ビルドツール）

**これは何か：**  
JavaプロジェクトのLibrary（外部の部品）を自動でダウンロードし、コードを動かせる形（`.jar`ファイル）にまとめてくれる「管理ツール」です。

> **料理の例え：**  
> Mavenは「材料の買い物リスト管理係」です。  
> `pom.xml`（設定ファイル）に「Spring Boot が必要」「PostgreSQLドライバーが必要」と書いておくと、インターネットから自動でダウンロードしてくれます。

**GradleではなくMavenを選んだ理由：**

| 観点 | Maven | Gradle |
|---|---|---|
| 設定ファイル | `pom.xml`（XML形式・読みやすい） | `build.gradle`（プログラム形式・難しい） |
| 学習コスト | 低い | やや高い |
| 日本の企業での普及率 | 非常に高い | モダン系スタートアップが中心 |
| 日本語情報量 | 豊富 | 英語情報が中心 |

---

### 2-4. Spring Data JPA + Hibernate（データベース操作）

**これは何か：**  
JavaのクラスとデータベースのテーブルをつなぐORM（Object-Relational Mapping）の仕組みです。

> **「ORM」とは**  
> 通常、データベースを操作するには SQL を書く必要があります：  
> ```sql
> INSERT INTO tasks (title, priority) VALUES ('買い物', 'high');
> ```
> ORMを使うと、SQL を書かずにJavaのコードだけで同じことができます：  
> ```java
> taskRepository.save(new Task("買い物", "high"));
> ```

**2つの関係：**
- **JPA**：「こういう操作ができる」という設計図（インターフェース）
- **Hibernate**：JPAの設計図を実際に動かす部品。Spring Boot が自動で使ってくれる

**なぜJPAか：**  
Spring Boot の標準セットに含まれており、追加の学習コストなしで使えます。  
`JpaRepository` を継承するだけで `save()`・`findAll()`・`deleteById()` などのメソッドが自動で使えます。

---

### 2-5. Flyway（データベースマイグレーション）

**これは何か：**  
データベースの構造（テーブルの作成・変更）をSQLファイルで管理し、変更履歴を追跡するツールです。

> **Gitの例え：**  
> コードはGitでバージョン管理しますが、データベースの構造もFlywayで「バージョン管理」できます。  
> `V1__create_tasks_table.sql` → `V2__add_column_to_tasks.sql` のように番号で管理します。

**使い方のイメージ：**
```
src/main/resources/db/migration/
├── V1__create_tasks_table.sql   ← 最初にtasksテーブルを作る
└── V2__insert_sample_data.sql   ← サンプルデータを入れる（任意）
```
アプリを起動するだけで、Flywayが未適用のSQLを自動で実行してくれます。

**LiquibaseではなくFlywayを選んだ理由：**  
FlywayはSQLをそのまま書くだけで使えます。Liquibaseは独自の記法を覚える必要があり、初心者には複雑です。

---

### 2-6. SpringDoc OpenAPI（Swagger UI）

**これは何か：**  
バックエンドのAPI仕様書を自動生成し、ブラウザ上でAPIの確認・テストができるツールです。

**できること：**
- `http://localhost:8080/swagger-ui.html` を開くと全APIが一覧表示される
- ブラウザ上からAPIを実際に呼び出してテストできる（画面なしでバックエンドのテストが可能）
- フロントエンドとバックエンドを分担開発するときの「仕様共有ツール」になる

**導入方法：** `pom.xml` に1行追加するだけで自動で有効になります。

## 3. フロントエンドの採用技術

### 3-1. React 18.x（UIライブラリ）

**これは何か：**  
画面（UI）を「コンポーネント（部品）」として作るためのJavaScriptライブラリです。

> **レゴブロックの例え：**  
> 画面をレゴブロックのように「ヘッダー部品」「カラム部品」「カード部品」に分けて作り、それを組み合わせて1つの画面を作ります。

**v1.0との違い：**

| 項目 | v1.0（Vanilla JS） | v2.0（React） |
|---|---|---|
| 画面の更新 | `innerHTML` で手動書き換え | Reactが差分だけ自動更新 |
| コードの構成 | 1つの `app.js` にすべて書く | コンポーネントに分割して管理 |
| データ管理 | 変数で直接管理 | `useState` フックで状態管理 |

**コンポーネント構成のイメージ：**
```
App.jsx
└── Board.jsx（ボード全体）
    ├── Column.jsx（「やること」カラム）
    │   ├── TaskCard.jsx（タスクカード）
    │   └── TaskCard.jsx
    ├── Column.jsx（「進行中」カラム）
    └── Column.jsx（「完了」カラム）
        └── TaskFormModal.jsx（追加・編集フォーム）
```

---

### 3-2. Vite 5.x（フロントエンドビルドツール）

**これは何か：**  
Reactアプリの開発を支援するツールです。  
コードを書くと即座に画面に反映される「ホットリロード（自動更新）」や、本番環境用のファイル生成（ビルド）を担当します。

**Create React App（CRA）ではなくViteを選んだ理由：**

| 観点 | Vite | Create React App（CRA） |
|---|---|---|
| 起動速度 | 非常に速い（数秒） | 遅い（数十秒） |
| 画面の更新速度 | ほぼ即時 | 数秒かかることがある |
| 現在の状況 | 業界標準（2024年〜） | **公式で非推奨（2023年〜）** |

> **重要：** Create React App は2023年にFacebook（Meta）の公式リポジトリで「非推奨」とされました。新規プロジェクトにはViteを使うのが現在の標準です。

---

### 3-3. Tailwind CSS 3.x（CSSフレームワーク）

**これは何か：**  
HTMLのクラス名を書くだけでデザインが適用される「ユーティリティファーストCSSフレームワーク」です。

**使い方のイメージ：**
```jsx
// Tailwind CSS を使った場合
<div className="flex items-center gap-4 bg-blue-500 text-white p-3 rounded-lg">
  タスクカード
</div>
```
クラス名の意味：`flex`（横並び）、`bg-blue-500`（青背景）、`p-3`（余白）、`rounded-lg`（角丸）

**他の選択肢との比較：**

| 観点 | Tailwind CSS | Material UI（MUI） | 素のCSS |
|---|---|---|---|
| 学習コスト | 低〜中 | 中（コンポーネントAPIを覚える） | 低（今まで通り） |
| カスタマイズ性 | 非常に高い | Googleデザインに縛られやすい | 完全自由 |
| 就職市場での需要 | 急速に増加中 | 企業でよく使われる | 基礎として評価される |
| v1.0からの移行しやすさ | ◎（CSS変数の概念が活かせる） | △（設計思想が変わる） | ◎（そのまま使える） |

**Tailwindを選んだ理由：**  
クラス名だけでスタイルが完結するため、CSSファイルとHTMLを行き来する必要がありません。  
日本のフロントエンド求人での言及が急増しており、就職に直結するスキルです。

---

### 3-4. Axios 1.x（HTTPクライアント）

**これは何か：**  
フロントエンド（React）からバックエンド（Spring Boot）のAPIに通信するためのライブラリです。

> **郵便屋さんの例え：**  
> Axiosは「郵便屋さん」です。  
> Reactが「タスクを保存して！」というリクエストを封筒に入れると、AxiosがSpring Bootのサーバーに届け、返事（レスポンス）を持ち帰ってきます。

**Fetch APIではなくAxiosを選んだ理由：**

| 観点 | Axios | Fetch API（ブラウザ標準） |
|---|---|---|
| HTTPエラーの処理 | 400・404・500エラーを自動で `catch` できる | 手動で `response.ok` を確認が必要 |
| JSONの変換 | 自動（`res.data` で取得できる） | 手動（`response.json()` を呼ぶ必要がある） |
| コードの読みやすさ | 高い（記述が短くシンプル） | やや低い（定型文が多い） |
| インストール | `npm install axios` の1行 | 不要（ブラウザ内蔵） |

---

### 3-5. useState + useContext（状態管理）

**これは何か：**  
Reactコンポーネント間でデータを共有する仕組みです（React標準機能）。

> **「状態（State）」とは**  
> 画面に表示されているデータのことです。例えば「現在のタスク一覧」や「モーダルが開いているかどうか」が「状態」です。

**仕組みのイメージ：**
```
TaskContext（タスク一覧を全体で共有）
├── Board.jsx が「タスク一覧を取得して！」と呼ぶ
├── TaskCard.jsx が「このタスクを削除して！」と呼ぶ
└── TaskFormModal.jsx が「新しいタスクを追加して！」と呼ぶ
```

**ReduxやZustandではなくuseContextを選んだ理由：**

| 観点 | useState + useContext | Zustand | Redux |
|---|---|---|---|
| インストール | 不要（React標準） | 1行で追加 | 複数パッケージが必要 |
| 学習コスト | 低い（React公式チュートリアルの範囲） | 低〜中 | 高い（概念が多い） |
| このアプリへの適性 | 十分（1画面・1データ種類） | やや過剰 | 完全に過剰 |

> **ポイント：** KanbanボードのようなシンプルなアプリにReduxを使うのは「大砲でハエを叩く」ようなものです。  
> まずReact標準機能をマスターしてから、大規模プロジェクトで必要になったときにZustand/Reduxを学ぶ順序が最適です。

## 4. インフラ・環境設定

### 4-1. Docker + docker-compose（開発環境のDB起動）

**これは何か：**  
アプリを「コンテナ」という隔離された箱の中で動かす仕組みです。  
PostgreSQLをMac/Windows に直接インストールせず、コンテナ内で起動できます。

> **引越し用コンテナの例え：**  
> 引越しのとき、家具をコンテナに入れて運ぶと、どこに運んでも同じ状態で使えます。  
> Dockerも同様に「アプリに必要なものをコンテナに入れる」ことで、どのPCでも同じ環境を再現できます。

**ローカル直接インストールではなくDockerを選んだ理由：**

| 観点 | Docker | ローカル直接インストール |
|---|---|---|
| PCへの影響 | なし（コンテナ内だけで動く） | PostgreSQLの設定がPCに残る |
| バージョン管理 | `docker-compose.yml` で固定できる | 手動で管理が必要 |
| 削除・やり直し | `docker-compose down -v` で完全削除 | アンインストールが手間 |
| チーム開発 | 全員が全く同じDB環境を使える | 「自分のPCだけ動く」問題が起きる |
| 就職後の実務 | 必須スキル（ほぼすべての現場で使用） | - |

**`docker-compose.yml` のイメージ：**
```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: taskboard
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    ports:
      - "5432:5432"
```
これだけで `docker-compose up -d` の1コマンドでPostgreSQLが起動します。

---

### 4-2. .env ファイル（環境変数管理）

**これは何か：**  
データベースのパスワードやAPIのURLなど「環境によって変わる値」を、コードに直書きせず別ファイルで管理する仕組みです。

**なぜ必要か：**  
コードにパスワードを直接書いてGitHubにアップロードすると、パスワードが世界中に公開されてしまいます。  
`.env` ファイルは `.gitignore` に追加してGitにコミットしないことで安全に管理できます。

**バックエンド（Spring Boot）の例：**
```properties
# application.properties
spring.datasource.url=jdbc:postgresql://localhost:5432/taskboard
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
```
```
# .env（Gitにコミットしない）
DB_USER=user
DB_PASSWORD=password
```

**フロントエンド（React + Vite）の例：**
```
# .env（Gitにコミットしない）
VITE_API_BASE_URL=http://localhost:8080
```
```js
// コード内での使い方
const API_URL = import.meta.env.VITE_API_BASE_URL;
```

## 6. 不採用技術とその理由

| 技術名 | 不採用の理由 |
|---|---|
| Next.js | 今回はReact単体で十分。Next.jsはSSR（サーバーサイドレンダリング）が必要な場合に使う |
| TypeScript | 今回は学習コスト削減のためJavaScriptを使用。将来の拡張時に移行を検討する |
| Redux / Zustand | このアプリの規模には過剰。React標準のuseContextで十分 |
| Create React App | 2023年に公式で非推奨となったため不採用 |
| Liquibase | FlywayよりSQLの書き方が複雑。初心者にはFlywayが適切 |
| MySQL / SQLite | 今回はPostgreSQLを指定。PostgreSQLは型が豊富で本番環境でも多く使われる |
| localStorage | サーバーへの移行により不要。ブラウザを閉じてもデータが消えない永続化が実現できる |

## 7. 外部ライブラリ・依存関係

### バックエンド（pom.xml に追加するライブラリ）

| ライブラリ名 | 役割 |
|---|---|
| spring-boot-starter-web | HTTPリクエストを受け取るWebサーバー機能 |
| spring-boot-starter-data-jpa | データベース操作（JPA/Hibernate） |
| spring-boot-starter-validation | フォームの入力チェック機能 |
| postgresql | JavaとPostgreSQLをつなぐドライバー |
| flyway-core | データベースのバージョン管理 |
| springdoc-openapi-starter-webmvc-ui | Swagger UI（API仕様書の自動生成） |
| lombok | `@Getter`/`@Setter` などの定型コードを自動生成 |
| spring-boot-devtools | コード変更時のサーバー自動再起動 |

### フロントエンド（npm でインストールするライブラリ）

| ライブラリ名 | 役割 |
|---|---|
| react, react-dom | Reactの本体 |
| axios | APIとの HTTP通信 |
| tailwindcss, postcss, autoprefixer | Tailwind CSS のインストールに必要なセット |

> **補足：** v1.0の「外部ライブラリなし」方針から、本格的なWebアプリ開発に必要な最小限のライブラリ構成に移行します。
