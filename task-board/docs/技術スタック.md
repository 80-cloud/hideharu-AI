## 1. フルスタック構成の全体像

### アプリ全体の構造

```
┌─────────────────────────────────────┐
│         ブラウザ（ユーザーの画面）         │
│                                     │
│   React + Tailwind CSS（フロントエンド）   │
│   ・タスクの表示・操作画面                 │
└──────────────┬──────────────────────┘
               │  HTTP通信（Axios）
               ▼
┌─────────────────────────────────────┐
│     Spring Boot サーバー（バックエンド）    │
│                                     │
│   Java + Gradle で動くWebサーバー        │
│   ・ブラウザからのリクエストを受け取る         │
│   ・タスクの追加・更新・削除の処理をする       │
└──────────────┬──────────────────────┘
               │  JPA/Hibernate（SQL自動生成）
               ▼
┌─────────────────────────────────────┐
│       PostgreSQL（データベース）          │
│                                     │
│   タスクのデータを永続的に保存               │
│   Docker コンテナ内で動作                │
└─────────────────────────────────────┘
```

### APIエンドポイント設計

フロントエンドとバックエンドは REST API で通信します。

| HTTPメソッド | URL | 操作 | v1.0の対応関数 |
|---|---|---|---|
| GET | `/api/tasks` | タスク一覧を取得 | `loadTasks()` |
| POST | `/api/tasks` | タスクを新規作成 | `addTask()` |
| PUT | `/api/tasks/{id}` | タスクを更新（編集・カラム移動） | `updateTask()` |
| DELETE | `/api/tasks/{id}` | タスクを削除 | `deleteTask()` |

## 4. インフラ・環境設定

### 4-1. Docker + docker-compose（開発環境のDB起動）

**`docker-compose.yml` の構成：**
```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: taskboard
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    ports:
      - "5432:5432"
```

---

### 4-2. .env ファイル（環境変数管理）

パスワードやURLなどの機密情報をコードに直書きせず、`.gitignore` で除外管理します。

**バックエンド（Spring Boot）：**
```yaml
# application.yml
spring:
  datasource:
    url: ${DATABASE_URL:jdbc:postgresql://localhost:5432/taskboard}
    username: ${DATABASE_USERNAME:taskboard}
    password: ${DATABASE_PASSWORD:taskboard}
```
- `${VAR:default}` 形式で、環境変数が無ければ default を使用
- ローカル開発：default のまま docker-compose の DB に接続
- AWS 本番：`~/.env` の env-var が default を上書きして RDS に接続

**フロントエンド（React + Vite）：**
```
# .env
VITE_API_BASE_URL=http://localhost:8080
```
```js
const API_URL = import.meta.env.VITE_API_BASE_URL;
```

---

### 4-3. AWS（クラウドデプロイ環境）

ローカル開発に加え、**AWS 無料枠** で本格的なデプロイを学習・実施しています。
詳細な構成・設計方針は [`インフラ構成.md`](インフラ構成.md) 参照。

**採用サービス：**

| サービス | 用途 | 採用理由 |
|---|---|---|
| **EC2 (t3.micro)** | アプリサーバー | 学習で各レイヤーを観察可能、無料枠 750h/月 |
| **EBS (gp3)** | EC2 ストレージ | 暗号化対応、デフォルト選択肢 |
| **VPC + Subnet + IGW + RT** | カスタムネットワーク | IaC 学習効果を最大化 |
| **Security Group** | ファイアウォール | リソース単位、SG参照で動的に絞り込み可能 |
| **RDS (PostgreSQL 16, db.t3.micro)** | マネージドDB | バックアップ・パッチを AWS 任せ、無料枠対象 |
| **Elastic IP** | EC2 の固定IP | 再作成時もIP不変 |
| **AWS Budgets** | 課金アラート | 1USD で即通知（事故防止） |

**ローカル開発との関係：**

| 環境 | DB | 起動コマンド |
|---|---|---|
| ローカル | docker-compose の Postgres (`localhost:5432`) | `docker compose up -d` + `gradlew bootRun` |
| AWS | RDS PostgreSQL (`*.rds.amazonaws.com:5432`) | EC2 上で `~/start.sh` |

`application.yml` の env-var 機構により、**同一コードがローカル・AWS両環境で動作** します。

---

### 4-4. Terraform（IaC ツール）

AWS リソースを宣言的に管理するためのツール（HashiCorp 製）。

**採用理由：**
- マネジメントコンソールでの手作業を排除（再現性確保）
- `terraform plan` で **適用前に差分を確認** できる
- `terraform destroy` で **撤収を確実に** 実行できる（無料枠超過事故の防止）
- Git でインフラの変更履歴を管理できる

**採用ツール群：**

| ツール | 役割 |
|---|---|
| **Terraform** | インフラの宣言的管理（メイン） |
| **AWS CLI** | 認証設定・確認・補助操作 |
| **tflint** | Terraform 構文・ベストプラクティスチェック |
| **tfsec** | Terraform セキュリティスキャン |
| **shellcheck** | cloud-init / 起動スクリプトの品質チェック |

品質チェックは `~/.claude/skills/terraform-quality-check/` スキルにまとめてあります。

実コードは [`infra/`](../../infra/) 配下を参照。

---

### 4-5. cloud-init / user_data（EC2 初期化）

EC2 起動時に **1回だけ** 自動実行されるスクリプト。

**user_data でやること：**
- 4GB swap 作成（t3.micro メモリ不足対策）
- パッケージ更新（dnf update）
- Java 25（Amazon Corretto）/ Node.js 22 / Docker / Docker Compose / Git のインストール
- アプリリポジトリの clone
- `~/start.sh` の配置（実行権限付与）

これにより、`terraform apply` 実行から数分後には **アプリが動かせる状態の EC2** が立ち上がる。

実コードは [`infra/user_data.sh`](../../infra/user_data.sh) 参照。

## 6. 不採用技術とその理由

| 技術名 | 不採用の理由 |
|---|---|
| Next.js | React単体で十分。SSR（サーバーサイドレンダリング）が必要な場合に採用を検討する |
| TypeScript | 今回は学習コスト削減のためJavaScriptを使用。将来の拡張時に移行を検討する |
| Redux / Zustand | このアプリの規模には過剰。React標準のuseContextで十分 |
| Create React App | 2023年に公式で非推奨となったため不採用 |
| Liquibase | FlywayよりSQLの書き方が複雑 |
| MySQL / SQLite | PostgreSQLを選定。型が豊富で本番環境でも広く使われている |
| localStorage | サーバーへの移行により不要 |
