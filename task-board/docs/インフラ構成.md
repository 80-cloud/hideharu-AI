## 2. 採用クラウドサービス

| サービス | 用途 | 採用理由 |
|---|---|---|
| **EC2** | アプリサーバー | 計算リソースの基本ブロック。学習段階で全レイヤーを観察できる |
| **EBS（gp3）** | EC2のストレージ | デフォルト選択肢。暗号化を有効化 |
| **Elastic IP** | EC2 の固定IP | EC2 を再作成してもIP不変、SG変更時の手戻り削減 |
| **VPC + Subnet + IGW + Route Table** | ネットワーク基盤 | カスタムVPC で IaC 学習効果を最大化 |
| **Security Group** | ファイアウォール | リソース単位 / stateful。SG参照（IPなし）で動的に絞り込み |
| **RDS（PostgreSQL）** | マネージドDB | バックアップ・パッチ・スケールを AWS に任せる |
| **DB Subnet Group** | RDS配置の集合 | RDS は最低2AZ必須のため必要 |
| **Key Pair** | SSH鍵管理 | 公開鍵をAWSに登録し、秘密鍵はローカル保持 |
| **AWS Budgets** | 課金アラート | 1USD 超過で即メール通知（事故防止） |

## 3. ネットワーク設計

### サブネット配置

| 種別 | AZ | 用途 |
|---|---|---|
| パブリックサブネット A | ap-northeast-1a | EC2 配置 |
| パブリックサブネット B | ap-northeast-1c | RDS の DB Subnet Group 用（最低2AZ要件のため） |

**設計方針：**
- 両サブネットとも IGW にルートを持つ「パブリック」だが、RDS は `publicly_accessible=false` で実際にはパブリック到達不可
- プライベートサブネット + NAT Gateway 構成は **NAT Gateway 課金（月$30〜）** が無料枠を超えるため不採用
- 学習用途として「セミ・パブリック構成」を選択（理由は本書セクション7参照）

### 経路（Route Table）

- パブリックサブネット → 既定経路 `0.0.0.0/0` → IGW
- VPC 内通信（EC2 → RDS）は内部ルーティングで完結

## 4. セキュリティ設計（多層防御）

「**EC2 以外からは RDS に到達できない**」を **3層** で保証する設計。

| 層 | 仕組み | 内容 |
|---|---|---|
| 1. ネットワーク層 | VPCプライベート空間 | RDS のDNSは VPC 内プライベートIP（10.0.x.x）にしか解決しない |
| 2. リソース層 | `publicly_accessible=false` | AWS が RDS にパブリック経路を作らない |
| 3. ファイアウォール層 | Security Group 参照 | RDS SG の ingress は **EC2 SG の所属インスタンス** のみ許可（IP ベースではなくSG ID 参照） |

### EC2 側 SG（外向きの開放）

| ポート | 用途 | 許可元 |
|---|---|---|
| 22 | SSH | **自PC IP /32**（環境変数 `my_ip_cidr`）|
| 80 | HTTP（Nginx 学習チェック用） | **自PC IP /32** |
| 5173 | Vite dev server | **自PC IP /32** |
| 8080 | Spring Boot API | **自PC IP /32** |

すべて **自PC IP のみ** に絞り、不特定多数からのアクセスを遮断。

### EC2 側のホストセキュリティ

- **IMDSv2 強制**（メタデータエンドポイントのトークン必須化）
- **EBS gp3 暗号化**（保存時暗号化）
- **IAM ロール未アタッチ**（EC2からの AWS API 操作を遮断）
- SSH パスワード認証無効、公開鍵のみ
- ルートユーザーログインは `without-password`（鍵のみ）

### CORS / 認証情報

- アプリの `application.yml` は **環境変数駆動**（`${DATABASE_URL:default}` 形式）
- DB認証情報は EC2上の `~/.env`（パーミッション 600、Git管理外）から `start.sh` で注入
- 平文の `password = "..."` を Git にコミットしない（CLAUDE.md ルール7）

## 5. 運用フロー（高レベル）

### 構築（IaC）

```
Terraform で *.tf を編集
   ↓
terraform plan で差分プレビュー
   ↓
terraform apply で AWS に反映
   ↓
EC2 起動 → cloud-init (user_data) が自動実行
   - swap作成 / dnf update / Java25 / Node22 / Docker / Git
   - hideharu-AI を git clone
   - infra/start.sh を ~/start.sh にコピー
   ↓
SSH ログイン → ~/.env 作成（DB接続情報）
   ↓
~/start.sh 実行 → DB(Docker) / Backend / Frontend を起動
```

### 撤収

```
terraform destroy
   ↓
全 AWS リソース削除（state も空に）
```

### 日常運用

- 通常はマージ後の動作確認 → destroy で課金リスクゼロ
- インフラ変更は GitHub Issue → ブランチ → PR → マージ のワークフロー
- Terraform品質チェック（`~/.claude/skills/terraform-quality-check/` のスキル、Claude Code 環境固有）でコミット前検証

## 6. 無料枠ポリシー

学習段階のため **AWS 無料枠を超えない** ことを最重要事項とする。

### 無料枠サマリ（要点のみ、最新は [AWS公式](https://aws.amazon.com/jp/free/) 参照）

| サービス | 無料枠の大枠 | 期間 |
|---|---|---|
| EC2 (t3.micro / t4g.micro) | 750時間/月 | 新規アカウントから12ヶ月 |
| RDS (db.t3.micro) | 750時間/月、20GBストレージ | 12ヶ月 |
| EBS | 30GB/月 | 12ヶ月 |
| データ転送（外向き） | 100GB/月 | 常時無料 |
| Elastic IP | EC2 にアタッチして使用中なら無料 | 常時 |

### 無料枠を守るルール

- **インスタンスは1つだけ起動**（同時複数稼働は 750h/月 を超える）
- t2.micro は ap-northeast-1 で 2024年以降 Free Tier 対象外（**t3.micro を使う**）
- Multi-AZ / RDS Read Replica / Performance Insights は課金発生で **無効化**
- AWS Budgets で 1USD アラート（即時メール）
- 不要時は **`terraform destroy` で撤収**

## 7. IaC方針

### Terraform で管理する範囲

`infra/` 配下の `*.tf` ですべてのインフラを宣言：

| ファイル | 役割 |
|---|---|
| [`main.tf`](../../infra/main.tf) | リソース定義（VPC / Subnet / IGW / RT / SG / EC2 / EIP / RDS） |
| [`variables.tf`](../../infra/variables.tf) | 変数宣言（IP・ポート・インスタンスタイプ等） |
| [`outputs.tf`](../../infra/outputs.tf) | 出力（EIP / RDS endpoint / SSH / psql コマンド等） |
| [`user_data.sh`](../../infra/user_data.sh) | EC2 初回起動時の cloud-init スクリプト |
| [`start.sh`](../../infra/start.sh) | EC2 ログイン後にアプリを起動するスクリプト |
| [`terraform.tfvars.example`](../../infra/terraform.tfvars.example) | 値のサンプル（実値は `terraform.tfvars` に書き Git 管理外） |

### Terraform で **管理しない** 範囲

- `~/.env`（DB認証情報の手動配置 → 将来 SSM Parameter Store / Secrets Manager で自動化）
- 過去のインスタンス・スナップショット（destroy 時に手で消す or 残らない設定）
- 一時的な手動オペレーション（psql で直接データ確認等）

### 設計判断の根拠

- **「動かす」より「再現性」を優先**：マネジメントコンソールの手作業は禁止（手順を忘れる、再現できない）
- **詳細値はコードに、文書化は設計方針に**：CIDR や IP の具体値は `*.tf` を見れば分かるため、本書には載せない
- **段階的構築**：Phase 1 → 2 → ... と小さく刻んで、各段階で動作確認
